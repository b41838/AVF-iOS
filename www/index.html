Week 1 Discussion

1. Less obvious perks of using 'live' data resources include logins and payments. How could Twitter be used in these two scenarios?

It’s pretty obvious, the perks of using Twitter for logins, no one wants to create an account anymore and why should you?  Everyone that does anything that would need a login already has a Twitter, Facebook or Google account so why not use it for other things?  You can either auto fill forms with the data to create your account or straight up login using Twitter.

Using Twitter for payments is a little trickier.  I don’t really see the purpose for this but you can either send money to another user by using a hashtag or other services allow you to send money to other users via the follow feature, although there are some very dark gray areas about that in Twitters Terms of Service.

2. Visit Programmableweb.com and search for JSON-P data sources. Why would we need JSON-P, specifically?

JSONP allows you to gain access and retrieve data from external domain sources.  This allows developers to easily access public data hosted on someone else’s servers.  API’s like Instagram and Twitter are common examples but better examples are MLS services such as Zillow or Trulia who subscribe and grab data from countless local MLS services and display it on their own sites.

3. Based on the JSON-P search results: what data resource(s) do you think might pair well with which types of Native device features?

QRdvark will integrate well with the native camera to scan QR Codes.

Google Maps (or Google Places or Google Static Maps) will work good with geolocation.

I’m thinking about using Clickatell to manage the messages from the native contacts feature.

And that’s really it for the features I’m going to be using, not much for the accelerometer or compass that I can find.

4. Please describe any design and functionality intentions you may have for your Demo app components this month.

The main element for my project this month revolves around the camera so that’s one native function that is an absolute must, along with geolocation.  My app is going to be a QR scavenger hunt app so I will need to find and use a 3rd party QR code scanner to use in conjunction with my camera.

Also, I plan on restructuring my app using AJAX to get rid of a lot of the repetitive code.  Basically, my app will have the structure in one page then using AJAX suck the individual documents in the content area of the index page.  You’ve probably seen or thought of it before, it seems like the best and lightest weight way to code an app.

5. Briefly describe some past 'lessons learned' regarding workflow as it relates to research, design, development, etc.

The most important thing I’ve learned is to just get started.  It’s incredibly detrimental to stare at a blank canvas, so if you’re in an IDE, get your basic HTLM structure down or write out the names of your functions or simply make a bunch of comments.  It’s so much easier to get going when you have something started already, otherwise it’s almost like staring into the void.

Week 2 Discussion

1. What are some of the ways that you could change your HTML on your Demo app to be more accessible? (Remember, you can use the Section tag with jQuery, for instance)

One thing I really need to do better is to use alt tags and set up navigation to properly tab through.  The simplest way to keep your app accessible is to make sure that even if a user is using a screen reader, they can still navigate the page.  I always forget to put alt tags on images, which are key if you are trying duplicate the user experience for a person with visual impairments.  Having your navigation be the first thing parsed makes navigation much easier for people using a screen reader.

2. Visit a video service on the web (like YouTube) and search for "Screen Reader Demo" and "Mobile Screen Reader" - what are some of the common frustrations (or common support mechanisms) that you see among the examples that are given - how could you fix them (or use them)?

Screen reading technology has come a long ways in the last 10 years or so but unfortunately web and application developers have not.  The most common complaint is not the software but the inability for the software to work correctly.  Developers often forget (or choose to ignore) the things that I mention in my first answer which really makes it hard for some users to use a website or application.

3. Devices: Please list two devices (and some of their features) that are made specifically to support Accessibility.

Kindle Fire – I was just thinking about reading on the web then started thinking about reading on a tablet and eventually the Kindle popped into my head.  Like Google TalkBack and Apple VoiceOver, Kindle has a built in screen reader called IVONA which (obviously) turns text to speech.  Kindle also has the ability to use gestures to navigate to the Home screen, go back, access notifications, the local context menu and the accessibility settings.  You can also magnify the content, set font size, turn on closed captioning and toggle between mono and stereo if you can only hear out of one ear.

Firefox – Firefox actually has some pretty decent options for browsing the web.   Everyone knows about Ctrl+ and Ctrl- for zoom in / zoom out but you can also zoom the text only by going to the View menu and selecting Zoom > Zoom Text Only or you can install the Page Zoom Buttons add-on.  You can tell Firefox not to load images so that alt text is easier read by going to Tools > Options > Content and uncheck the “Load images automatically” box.  This is helpful for developers as well to make sure our alt tags are properly applied to everything.  You can alsoturn on the “Find As You Type” feature, which allows you to start typing after you load a web page and go directly to a set of key words.  Tools > Options > Advanced > General > Accessibility and mark the “Search for text when I start typing” box.  Finally you can use the Browse With the Caret feature to navigate through web pages the same as you would a word processor.  Enable/Disable this feature by pressing F7.

4. Mobile Apps: Please list two apps (either Android or iOS, or both) that are made specifically to support Accessibility.

LookTel (iOS only) – This is sweet!  I never thought about an app for this, it scans dollar bills and gives you the denomination and supports 21 different currencies.  It both reads you the denomination out loud and also gives you a large display with the option for high contrast mode.

Parking Mobility (iOS only) – An app designed to socially share the locations of handicapped parking spaces as well as report abuse of said parking spots.  This sounds pretty cool but I really wonder how many people use it and actually need to use it.  I use handicapped spots every day and I don’t know if I’ve ever had a problem finding them.  Finding an open one is a different story…  The reporting feature is a little lame though, it has no direct line to ticketing agencies rather just to local advocate groups.
