Week 1 Discussion

1. Less obvious perks of using 'live' data resources include logins and payments. How could Twitter be used in these two scenarios?

It’s pretty obvious, the perks of using Twitter for logins, no one wants to create an account anymore and why should you?  Everyone that does anything that would need a login already has a Twitter, Facebook or Google account so why not use it for other things?  You can either auto fill forms with the data to create your account or straight up login using Twitter.

Using Twitter for payments is a little trickier.  I don’t really see the purpose for this but you can either send money to another user by using a hashtag or other services allow you to send money to other users via the follow feature, although there are some very dark gray areas about that in Twitters Terms of Service.

2. Visit Programmableweb.com and search for JSON-P data sources. Why would we need JSON-P, specifically?

JSONP allows you to gain access and retrieve data from external domain sources.  This allows developers to easily access public data hosted on someone else’s servers.  API’s like Instagram and Twitter are common examples but better examples are MLS services such as Zillow or Trulia who subscribe and grab data from countless local MLS services and display it on their own sites.

3. Based on the JSON-P search results: what data resource(s) do you think might pair well with which types of Native device features?

QRdvark will integrate well with the native camera to scan QR Codes.

Google Maps (or Google Places or Google Static Maps) will work good with geolocation.

I’m thinking about using Clickatell to manage the messages from the native contacts feature.

And that’s really it for the features I’m going to be using, not much for the accelerometer or compass that I can find.

4. Please describe any design and functionality intentions you may have for your Demo app components this month.

The main element for my project this month revolves around the camera so that’s one native function that is an absolute must, along with geolocation.  My app is going to be a QR scavenger hunt app so I will need to find and use a 3rd party QR code scanner to use in conjunction with my camera.

Also, I plan on restructuring my app using AJAX to get rid of a lot of the repetitive code.  Basically, my app will have the structure in one page then using AJAX suck the individual documents in the content area of the index page.  You’ve probably seen or thought of it before, it seems like the best and lightest weight way to code an app.

5. Briefly describe some past 'lessons learned' regarding workflow as it relates to research, design, development, etc.

The most important thing I’ve learned is to just get started.  It’s incredibly detrimental to stare at a blank canvas, so if you’re in an IDE, get your basic HTLM structure down or write out the names of your functions or simply make a bunch of comments.  It’s so much easier to get going when you have something started already, otherwise it’s almost like staring into the void.
